# Context Propagation Standard for iFlot

**Status:** Proposed
**Date:** April 2026
**Owners:** Architecture Lead / Platform Engineering
**Related ADR:** ADR-006 — Observability Architecture, Telemetry Model, and Operational Standards for iFlot

---

## 1. Purpose

This document defines the standard for propagating technical and business correlation context across iFlot.

It applies to:

* `iflot-web`
* `iflot-api`
* future background jobs
* future asynchronous flows
* future internal or external service-to-service communication

The objective is to ensure that a single business flow can be traced consistently across:

* frontend events and spans
* backend requests and spans
* structured logs
* future retries, async steps, and multi-trace execution

---

## 2. Scope

This standard covers:

* which headers must be used
* who generates each identifier
* how identifiers are propagated
* how they appear in logs and spans
* how they behave in synchronous and asynchronous flows

This standard does **not** define:

* exact log JSON schema
* exact metric names
* SLO thresholds
* alert rules

Those are defined in separate observability documents.

---

## 3. Core Principle

iFlot uses **two complementary propagation models**:

### 3.1 Technical propagation

Used to follow a single execution path.

Primary identifiers:

* `trace_id`
* `span_id`

Transport standard:

* `traceparent`
* `tracestate` (when present)

This propagation follows the W3C Trace Context standard.

### 3.2 Business propagation

Used to follow a single business operation across multiple requests, retries, spans, traces, and asynchronous steps.

Primary identifier:

* `operation_id`

Transport header:

* `X-Operation-Id`

Optional external request identifier:

* `X-Correlation-ID`

---

## 4. Standard Headers

The following headers are part of the iFlot propagation standard.

| Header             | Required | Purpose                                                   |
| ------------------ | -------: | --------------------------------------------------------- |
| `traceparent`      |      Yes | W3C trace context propagation                             |
| `tracestate`       | Optional | W3C trace vendor-specific state                           |
| `X-Operation-Id`   |      Yes | Business operation correlation across requests and traces |
| `X-Correlation-ID` | Optional | External/support-facing request correlation               |
|                    |          |                                                           |

### 4.1 Header semantics

#### `traceparent`

Used by OpenTelemetry to propagate the current trace context.

#### `tracestate`

Optional companion header for trace context state.

#### `X-Operation-Id`

Stable identifier for the business flow.
It must survive:

* HTTP retries
* frontend resubmissions when intentionally linked to the same business operation
* asynchronous steps
* multi-service execution
* multiple traces belonging to the same business operation

#### `X-Correlation-ID`

Optional request-facing correlation ID.
It is useful for:

* support teams
* external integration partners
* manual log lookup

It does **not** replace `trace_id` or `operation_id`.

---

## 5. Identifier Definitions

## 5.1 `trace_id`

A technical identifier representing one distributed trace.

Properties:

* generated by OpenTelemetry tracing infrastructure
* may change between separate executions
* may differ across async boundaries if a new trace starts
* primary key for trace exploration

Use it for:

* trace navigation
* span correlation
* logs-to-trace linking

## 5.2 `span_id`

A technical identifier representing one specific span within a trace.

Use it for:

* precise span/log correlation
* lower-level diagnostics

## 5.3 `operation_id`

A business correlation identifier representing one business operation.

Properties:

* stable across the life of the business flow
* survives retries
* survives async steps
* can group multiple traces
* independent from entity IDs such as `trip_id`

Use it for:

* flow-level diagnosis
* grouping multiple traces into one business operation
* querying logs when a trace has split across steps

## 5.4 `X-Correlation-ID`

An external request correlation identifier.

Properties:

* may come from frontend, API consumers, gateway, or external systems
* may be generated by backend if absent
* reflected in response headers
* visible in logs

Use it for:

* support lookups
* external debugging
* request-level troubleshooting

---

## 6. Generation Rules

## 6.1 `trace_id`

Generated and managed by OpenTelemetry.

No application code should manually generate `trace_id` values.

## 6.2 `operation_id`

The generation rules are mandatory.

### Rule A — user-initiated flows

For flows initiated by the frontend:

* `iflot-web` generates `operation_id`
* it sends the value in `X-Operation-Id`
* `iflot-api` must preserve it for the entire flow

Examples:

* Login
* Create Trip
* Close Trip
* Close Guide
* Generate Pre-Invoice

### Rule B — system-initiated flows

For flows initiated without a frontend request:

* `iflot-api` or the initiating system component generates `operation_id`

Examples:

* scheduled jobs
* retry processors
* async consumers
* compensation handlers

### Rule C — preserve if already present

If a request already includes `X-Operation-Id`:

* do not overwrite it
* preserve and propagate it

### Rule D — do not derive from entity IDs

`operation_id` must **not** be replaced by:

* `trip_id`
* `guide_id`
* `preinvoice_id`

These identifiers may be added as context, but they are not substitutes.

Reason:

* the business operation may exist before the entity exists
* one operation may affect multiple entities
* one entity may be involved in multiple operations over time

## 6.3 `X-Correlation-ID`

### Rule A — preserve when provided

If inbound request contains `X-Correlation-ID`:

* preserve it
* expose it in response headers
* include it in logs

### Rule B — generate if absent

If not present:

* backend generates a new request correlation ID at the entry point
* response must include it

### Rule C — request-scoped

`X-Correlation-ID` is request-facing and request-scoped.

It may remain stable across retries if the caller preserves it, but that is not its main guarantee.

---

## 7. Format Rules

## 7.1 `operation_id`

Recommended format:

* UUID v4 or UUID v7

Requirements:

* globally unique
* opaque
* no embedded business meaning
* safe for logs and headers

Example:

```text
8e4f8d6f-88f0-48d4-b7ab-6c6522e4e3c5
```

## 7.2 `X-Correlation-ID`

Recommended format:

* UUID v4 or UUID v7
* alternatively prefixed request ID if needed for support readability

Example:

```text
req-0f78db3d-30a1-4cd0-a81c-0c5f1e16a2d8
```

---

## 8. Synchronous HTTP Flow Standard

## 8.1 Frontend → Backend

For every business flow initiated from the UI:

1. frontend creates `operation_id`
2. frontend starts or continues trace context
3. frontend sends:

   * `traceparent`
   * `tracestate` if present
   * `X-Operation-Id`
   * `X-Correlation-ID` if available and desired
4. backend preserves and logs all relevant identifiers

### Example request

```http
POST /api/trips HTTP/1.1
Host: api.iflot.local
traceparent: 00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01
tracestate: vendor=value
X-Operation-Id: 8e4f8d6f-88f0-48d4-b7ab-6c6522e4e3c5
X-Correlation-ID: req-0f78db3d-30a1-4cd0-a81c-0c5f1e16a2d8
Content-Type: application/json
```

### Example response

```http
HTTP/1.1 201 Created
X-Operation-Id: 8e4f8d6f-88f0-48d4-b7ab-6c6522e4e3c5
X-Correlation-ID: req-0f78db3d-30a1-4cd0-a81c-0c5f1e16a2d8
Content-Type: application/json
```

## 8.2 Backend → Downstream HTTP

When backend calls another service or internal component over HTTP:

* preserve `traceparent`
* preserve `tracestate` if present
* forward `X-Operation-Id`
* forward `X-Correlation-ID` if present

Rule:

> The backend must never invent a new `operation_id` for a downstream call that is part of the same business flow.

---

## 9. Asynchronous Flow Standard

This section defines behavior for future jobs, queues, events, and asynchronous workers.

## 9.1 General rule

When a synchronous flow triggers an asynchronous step:

* `operation_id` must always be propagated
* `traceparent` should be propagated when the transport supports it and when continuation is meaningful
* if a new trace starts, `operation_id` becomes the stable cross-trace link

## 9.2 Async message metadata

For events, queue messages, or job payload metadata, include:

* `operation_id`
* `traceparent` if available and supported
* `correlation_id` if relevant
* `flow_name`
* relevant domain identifiers if available

### Example event envelope

```json
{
  "metadata": {
    "operation_id": "8e4f8d6f-88f0-48d4-b7ab-6c6522e4e3c5",
    "traceparent": "00-4bf92f3577b34da6a3ce929d0e0e4736-00f067aa0ba902b7-01",
    "correlation_id": "req-0f78db3d-30a1-4cd0-a81c-0c5f1e16a2d8",
    "flow_name": "close_guide"
  },
  "payload": {
    "guide_id": "G-1001"
  }
}
```

## 9.3 New trace allowed

Consumers may create a new trace when processing async work.

If that happens:

* preserve `operation_id`
* log both `operation_id` and the new `trace_id`
* include domain context identifiers where possible

---

## 10. Logging Standard for Propagated Context

## 10.1 Required log context fields

The following fields must be present in logs when available:

* `trace_id`
* `span_id`
* `operation_id`
* `correlation_id`
* `flow_name`
* `tenant_id`

## 10.2 Backend MDC mapping

Backend must map propagated values into MDC for request handling and downstream logging.

Recommended MDC keys:

* `trace_id`
* `span_id`
* `operation_id`
* `correlation_id`
* `flow_name`
* `tenant_id`

## 10.3 Cleanup rule

MDC must be cleared at request completion and correctly re-established across async boundaries.

---

## 11. Frontend Implementation Rules

## 11.1 Frontend responsibilities

For user-initiated flows, frontend must:

* generate `operation_id`
* start or continue trace context using OpenTelemetry
* attach `X-Operation-Id` to API requests
* optionally attach `X-Correlation-ID` when required by policy
* log or emit frontend telemetry with:

  * `operation_id`
  * `trace_id` where available
  * `flow_name`

## 11.2 Frontend retry rule

If the UI retries the same logical business operation intentionally, it must preserve the same `operation_id`.

If the user starts a genuinely new business action, a new `operation_id` must be created.

### Examples

#### Same `operation_id`

* user clicks “Create Trip”
* transient network failure occurs
* UI retries automatically for the same submission

#### New `operation_id`

* user cancels previous attempt
* user edits the form and starts a new Create Trip action later

---

## 12. Backend Implementation Rules

## 12.1 Inbound HTTP filter/interceptor

Backend must implement an entry-point mechanism that:

* extracts `X-Operation-Id`
* extracts `X-Correlation-ID`
* generates missing `X-Operation-Id` only for system-initiated/backend-originated flows or when policy explicitly allows fallback at the API boundary
* generates missing `X-Correlation-ID`
* places values into MDC
* ensures response headers include:

  * `X-Operation-Id`
  * `X-Correlation-ID`

## 12.2 Outbound HTTP clients

All outbound HTTP clients must propagate:

* `traceparent`
* `tracestate` if present
* `X-Operation-Id`
* `X-Correlation-ID` if present

This applies to:

* RestTemplate
* RestClient
* WebClient
* future HTTP clients

## 12.3 Job / consumer startup

Jobs and async consumers must:

* extract `operation_id` from message metadata if present
* generate one only when the flow genuinely starts there
* establish logging context before business execution starts

---

## 13. Flow Name Rule

`flow_name` is required for all critical flows.

It must be:

* stable
* human-readable
* lower_snake_case
* shared across frontend and backend

Examples:

* `login`
* `create_trip`
* `close_trip`
* `close_guide`
* `generate_preinvoice`

Rule:

> Frontend and backend must use the same `flow_name` for the same business flow.

---

## 14. Decision Matrix

| Situation                     | trace_id                     | operation_id                 | X-Correlation-ID            |
| ----------------------------- | ---------------------------- | ---------------------------- | --------------------------- |
| Frontend starts user flow     | OTel/browser                 | frontend generates           | optional/preserve           |
| Backend receives HTTP request | preserve/extract             | preserve or generate by rule | preserve or generate        |
| Backend calls downstream HTTP | propagate                    | propagate                    | propagate if present        |
| Async event emitted           | optional propagate           | propagate                    | propagate if relevant       |
| Async consumer starts work    | preserve or create new trace | preserve                     | preserve if relevant        |
| Scheduled job starts flow     | create                       | backend generates            | backend generates if needed |

---

## 15. Anti-Patterns

The following are forbidden:

### 15.1 Replacing `operation_id` with entity IDs

Bad:

* using `trip_id` as `operation_id`
* using `guide_id` as `operation_id`

### 15.2 Generating a new `operation_id` in each layer

Bad:

* frontend generates one
* controller generates another
* service generates another

### 15.3 Using `X-Correlation-ID` as the only correlation mechanism

Bad because:

* it does not replace tracing
* it does not model business flow continuity like `operation_id`

### 15.4 Forgetting response headers

Bad:

* request enters with correlation context
* response does not return identifiers

### 15.5 High-cardinality metrics from propagated IDs

Bad:

* using `operation_id` as a metric label
* using `correlation_id` as a metric label

These identifiers belong in logs and traces, not in metrics.

---

## 16. Minimal Examples

## 16.1 Frontend request creation

```text
- user clicks "Close Guide"
- frontend creates operation_id
- frontend starts/continues trace
- frontend calls backend with traceparent + X-Operation-Id
```

## 16.2 Backend request handling

```text
- backend extracts trace context
- backend extracts X-Operation-Id
- backend generates X-Correlation-ID if absent
- backend places identifiers in MDC
- backend logs flow start
- backend executes business logic
- backend returns identifiers in response headers
```

## 16.3 Async continuation

```text
- backend emits event with operation_id + traceparent
- async consumer receives message
- consumer creates or continues trace depending on design
- consumer preserves operation_id
- logs and spans remain queryable by operation_id across traces
```

---

## 17. Acceptance Criteria

This standard is considered implemented when:

1. frontend sends `X-Operation-Id` for critical user-initiated flows
2. backend preserves and returns `X-Operation-Id`
3. backend preserves or generates `X-Correlation-ID` and returns it
4. backend logs contain `trace_id`, `span_id`, `operation_id`, and `correlation_id`
5. outbound backend HTTP calls propagate the required headers
6. async/event designs preserve `operation_id`
7. frontend and backend share the same `flow_name` values

---

## 18. Final Rule

In iFlot:

* `trace_id` is the primary **technical** correlation key
* `operation_id` is the primary **business flow** correlation key
* `X-Correlation-ID` is the primary **external request/support** correlation key

All three may coexist, but they serve different purposes and must not be conflated.
